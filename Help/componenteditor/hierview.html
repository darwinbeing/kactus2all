Hierarchical view editor

Hierarchical view editor is used to edit the details of a hierarchical view of the component. The top part of the editor is identical with hierarchical view. User may change the view type and then the bottom part changes accordingly.

View has name and textual description. Display name can be more detailed or user-friendly than name, but usually only few words.

Environment identifiers in mandatory textual information about how this view deployed in tool environment. Language indicates this view may be compatible with a particular tool, but only if that language is supported in that tool, e.g., different versions of some simulators may support two or more languages. Tool indicates that this view contains tool-information, e.g., HDL models that use simulator-specific extensions. The tool can have a generic value such as "*Simulation" and "*Synthesis"VendorSpecific can further qualify tool and language compatibility, e.g. if some appropriate simulator mode must be enabled.

View type can be changed and a non-hierarchical view has 2 additional settings.

  • Hierarchy reference is a mandatory VLNV-reference to a design configuration or design document. These documents list the sub-components instantiated in the hierarchical design and their configurations. User can simply drag-and-drop appropriate design from IP library over the fields, or type them. Kactus2 has also a graphical design editor where the user instantiate and configure components very easily.
  • Reference to a top-level implementation view is a Kactus2-specific extension. It is a reference to a non-hierarchical view of the same component that contains the generated RTL codes. A design in Kactus2 is graphical block diagram with sub-components and Kactus2 can automatically generate top-level structural VHDL from it. This extension ensures that generated codes are automatically included when generating a compilation script for simulation or synthesis.
